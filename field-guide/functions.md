# Functions

*Lesson 2*

**Keywords:** `function` · `def` · `define` · `call` · `parameter` · `argument` · `built-in` · `user-defined`

---

**YOU'LL HEAR THIS WHEN** *you're in a technical planning or architecture discussion*

A developer says: *"We should abstract that out into its own function"* or *"That logic shouldn't be duplicated."*

You might be a PM or technical program manager in a sprint planning session. They're applying DRY at the action level — consolidating repeated logic into one reusable block.

---

**DEV-TO-ENGLISH** *translation layer*

*"We need to refactor that"* → this code was written without enough functions. The same logic is scattered around in multiple places and needs to be consolidated. It's not broken — it's not built to last.

---

**QUESTIONS WORTH ASKING** *when a feature takes longer than expected*

→ "Is this because we need to build it from scratch, or can we reuse something that already exists?"

This is the function/reuse question in plain English. It signals genuine technical awareness from a non-technical person without overreaching.

---

**SO WHAT?** *the one-line reason this matters to you*

Functions are why software can scale. Every time someone talks about 'reusable components' or 'modular design,' they're describing the function principle at a larger level.
